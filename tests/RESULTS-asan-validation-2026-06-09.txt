(a) CANARY UAF POSITIVE CONTROL
══════════════════════════════════════════════════════════════════════════════════

CANARY DESIGN
-------------
- Location: src/export/mod.rs, export_loop(), immediately before the main loop{},
  after the "otel export: export loop started" log notice (line ~492).
- Trigger: file-gated on /tmp/asan-canary-uaf (not env-var, because nginx scrubs
  the env in child processes via ngx_set_environment; ASAN_OPTIONS works because
  the ASan runtime parses it at exec before the scrub — but a runtime
  std::env::var check in the exporter would silently not fire).
- Code (temporary, no_std crate — Box not in scope; uses libc malloc/free):
    if std::path::Path::new("/tmp/asan-canary-uaf").exists() {
        unsafe {
            let p = libc::malloc(64) as *mut u8;
            libc::free(p as *mut libc::c_void);
            // UAF: read from freed heap memory
            let v = std::ptr::read_volatile(p);
            std::hint::black_box(v);
        }
    }
- Placement rationale: export_loop() is an async fn spawned unconditionally by
  otel_exporter_cycle() for every nginx start; run_grpc_export.sh exercises the
  full export path including the first drain tick, so the canary site is
  guaranteed to execute on every harness invocation.
